@ViewChild & @ViewChildren — Easy Explanation

What is it?

@ViewChild → get ONE element from template
@ViewChildren → get ALL elements from template ====================================================

React equivalent:

React
const inputRef = useRef(null); // one element
No direct equivalent for multiple refs (need array of refs)
==============================================================

Angular equivalent:

// Angular
//@ViewChild('myInput') myInput!: ElementRef; // one element
//@ViewChildren('myItem') items!: QueryList<>; // multiple elements
==================================================================

Why Do We Need It?

Sometimes you need to:
→ Auto focus an input ✅
→ Get input value directly ✅
→ Access child component ✅
→ Scroll to element ✅
→ Init third party library ✅
(charts, maps, datepicker)
=======================================

When to Use?

Use @ViewChild when:
→ Need direct DOM access
→ Need to call child component method
→ Need to read element value
→ Third party lib needs DOM element
Always access in:
→ ngAfterViewInit ✅
→ NOT in constructor ❌
→ NOT in ngOnInit ❌ (not safe)
=====================================

@ViewChild — Two Ways to Use

Way 1

// Way 1 — Template Reference Variable (string)
// use when accessing DOM element
@ViewChild('myInput') myInput!: ElementRef;
//

Way 2

// Way 2 — Component Class (type)
// use when accessing child component
@ViewChild(ChildComponent) child!: ChildComponent;
===========================================================

What is ng-content? — Easy Explanation

ng-content = React's children prop

It lets you pass HTML from parent into child component.

React vs Angular — Direct Comparison

Why Do We Need It?

Without ng-content — rigid component:
<app-card></app-card>
→ always shows same content 😫
With ng-content — flexible component:
<app-card>Any content here!</app-card> <br>
→ reusable with different content ✅
========================================
Projected Content = whatever HTML you write
BETWEEN component's opening
and closing tags

which then gets displayed
wherever sits
inside that component
=================================================

Why Called "Projected"?

Think of it like a movie projector 🎥

ng-content = the projector screen (placeholder)
Projected content = the actual movie/image being shown on screen

Parent's HTML = the film reel
= the screen where it gets displayed

@ContentChild & @ContentChildren — Why, What, How, When

What is it?

@ContentChild → get ONE projected element/component (from ng-content)
@ContentChildren → get ALL projected elements/components (from ng-content)

Life cycle hooks for above both is --- AfterContentInit

@ViewChild vs @ContentChild — Key Difference

@ViewChild → element from component's OWN template
@ContentChild → element PASSED IN via ng-content (projected content)
